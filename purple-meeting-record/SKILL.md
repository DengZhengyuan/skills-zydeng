# purple-meeting-record

## 概述

`purple-meeting-record` 用于把嘈杂、弱结构甚至 speaker 标注不可靠的沟通转写，整理成 Purple Sys 可直接使用的记录。

它的目标不是生成一篇“像会议纪要的长文”，而是：

- 先把 transcript 压成可筛选的原子候选项
- 再把用户确认过的关键内容沉成一套完整稿
- 最后在用户明确允许后，把 meeting note、daily 和相关 log 内容自动写回 Purple Sys

这个 skill 默认采用四阶段交互，而不是一步到位：

1. `AI 初步清洗 + 归属判断 + 候选清单`
2. `用户修正归属并选择保留项`
3. `AI 生成完整稿包（meeting note + daily + log 追加内容）`
4. `用户明确说可以写入 -> AI 自动写入 Purple Sys`

## 核心规则

- 把 `meeting` 视为单次正式记录，先保内容，再补归属。
- 不把 transcript 原样改写成流水账。
- 不因为用户提供了 transcript，就默认知道全部元信息；先自行推断，再补问少量关键问题。
- 不设默认 `整理档位`；若用户未明确给出档位，也没有用自然语言表达长短要求，必须先追问。
- 不把寒暄、设备调试、网络噪音、重复确认默认写入正式记录。
- 不把未经确认的关键判断强行挂到具体人名下。
- 不自动新建 student log 或 project log。
- 用户确认条目后，先展示一套完整稿包，而不是边确认边直接落库。
- 用户明确说“可以写入”“写入吧”“落库吧”等之后，默认把本轮已确认的 meeting note、daily 和相关既有 log 一并写入。
- 如果用户没有明确授权写入，默认停留在草稿阶段。
- “会议重要”只能提醒可能需要更高保真整理，不能替代 `整理档位` 选择。

## 适用模式

在开始整理前，先确定模式。若用户未明确说明，再根据 transcript 形态给出推荐，但不要直接替用户决定。

### 按人模式

适用于：

- 老师依次和不同学生沟通
- 多人轮流汇报近况
- 几位老师分别点评不同人的工作

按人模式的组织原则：

- 先给一段总体判断
- 主体按人分组
- 每个人只保留真正改变后续判断的内容

### 按事模式

适用于：

- 围绕一个主题或少数议题展开的讨论
- 电话沟通某件事情
- 与合作方或老师就某个问题集中确认

按事模式的组织原则：

- 先给一段总体判断
- 主体按事项分组
- 每个事项只保留推进状态、关键判断、决策和动作

## 整理档位 / 压缩强度

在生成正式整理结果前，必须锁定 `整理档位`。它描述的是信息保留量与压缩强度，不是会议重要性标签。

- 不设默认档位。
- 用户没有明确给出档位，且也没有用自然语言明确表达长短要求时，必须先问，再继续生成正式整理结果。
- 第一阶段可以先做基础清洗和候选清单，但进入第三阶段前必须锁定档位。
- 若用户只说“这是个很重要的会”，仍然信息不足，必须继续追问档位。

四级定义如下：

### 全细节

- 尽量保留全部有效信息点、判断变化、动作安排和必要语境。
- 不主动上升为概括性结论，不把丰富讨论压成短纪要。
- 允许做降噪、去口头语、合并纯重复内容，但不是 transcript 逐句誊写。

### 详细

- 保留绝大多数有效信息，允许适度归并和结构化整理。
- 可以压缩来回反复的过程性表述，但仍以完整保留为主。
- 适合“细一点，但不要脏流水账”的场景。

### 标准

- 保留关键判断、决策、动作、节点变化和必要背景。
- 省略大部分过程性往复和支撑细节。
- 适合常规可回看、可回填的 meeting note。

### 极简

- 只保留主题、结论、下一步和少量必要未决事项。
- 输出应明显短于其他档位，可压到很短的摘要级别。
- 适合用户只想快速记住“这次聊了什么、定了什么、接下来做什么”。

常见自然语言可直接映射为档位：

- “不要漏任何细节”“每一句都重要”“尽量全保留” -> `全细节`
- “详细一点”“写细一点”“保留大部分讨论内容” -> `详细`
- “正常整理”“按常规 meeting note 写” -> `标准`
- “写短一点”“两三百字”“只要结论和动作” -> `极简`

如果用户没给出足够明确的表达，使用固定追问模板：

- `这次你要哪种整理档位：全细节、详细、标准、极简？`
- 如需补一行说明，可写：`全细节=尽量不漏信息；详细=保留大多数有效内容；标准=只留关键判断/动作；极简=只留主题、结论和下一步。`

## 元信息获取顺序

默认按以下顺序获取元信息：

1. 从文件名、路径、文本开头、已有上下文推断
2. 从 transcript 本身抽取候选信息
3. 只对仍缺失且影响结果质量的字段向用户提问

默认优先推断：

- 标题
- 日期
- 参会人候选
- 沟通模式

`整理档位` 的处理规则：

- 不默认推断，也不因“会议重要”自动提高档位。
- 若用户已经用自然语言明确表达长短要求，直接映射到对应档位，不必重复追问。
- 若档位仍不明确，必须向用户追问。
- 未锁定档位前，不要直接进入第三阶段完整稿包。

默认只在必要时追问：

- `按人 / 按事`
- `整理档位`
- 会议标题，或用户是否接受 AI 推断标题
- 日期 / 时间
- 关键参与人或 speaker 映射

默认不阻塞输出：

- 地点 / 形式
- 完整参会人名单
- `related_projects`
- `related_people`

如果 transcript 文件名已经能稳定提供日期或标题，直接采用，并只在存在明显冲突时向用户确认。

## 第一阶段：生成工作底稿

第一阶段不要直接生成正式记录。先输出工作底稿，固定包含以下四块：

1. `元信息草稿`
2. `归属待确认`
3. `候选要点清单`
4. `建议保留项`

输出格式与写法契约见 [references/output-contract.md](references/output-contract.md)。

### 元信息草稿

写出当前可确定或可推断的关键信息：

- 整理档位
- 模式
- 标题
- 日期 / 时间
- transcript 质量判断
- 参会人候选

如果某项不确定，明确标注“待确认”，不要假装确定。

若用户尚未给出 `整理档位`，在元信息草稿中明确写出 `整理档位：待确认`，并在第一阶段结束时补问。

### 归属待确认

当 speaker 标签不可靠、一个账号代表多人、或某段话无法稳定归属时：

- 先给出 AI 的初步判断
- 把不确定片段归入“未确认发言 / 多人讨论”
- 标出哪些归属会显著影响最终记录
- 等用户最终裁决

未经确认，不要把关键决定、判断或动作绑定到某个人名下。

### 候选要点清单

候选要点必须满足：

- 一条只表达一个事实、判断、动作或必要背景
- 尽量压成一句，不写长段
- 优先保留能改变后续记录价值的信息

固定标签只有四类：

- `[进展]`
- `[判断]`
- `[动作]`
- `[背景]`

编号规则固定为：

- `按人模式`：`P1.1 / P1.2 / P2.1 ...`
- `按事模式`：`T1.1 / T1.2 / T2.1 ...`

候选清单不是 transcript 摘抄。应先降噪、合并重复信息，再输出原子项。

### 建议保留项

在候选清单后，主动给出一组建议保留编号，减少用户逐条筛选的负担。

默认优先建议保留：

- 会改变下一步动作的内容
- 会改变对某人 / 某事判断的内容
- 会影响长期跟踪的节点变化
- 少量必要背景

默认不建议保留：

- 寒暄
- 无效重复
- 设备 / 网络调试噪音
- 只帮助现场交流、但对事后记录无价值的细节

## 第二阶段：读取用户裁决

默认期望用户只回复四类内容：

- 归属修正
- 保留 / 删除的编号
- 整理档位指定
- 少量额外要求

例如：

- “P2.1 归到张舒瑞，不是姚冬旭”
- “保留 P1.1 P1.3 P2.1，删除 P1.2”
- “这次按详细写”
- “把老师的总判断写得更短”
- “给张舒瑞和姚冬旭补 student log 草稿”

如果用户只给出保留编号，没有逐条说明删除项，则默认：

- 保留用户点名的编号
- 同时保留与之强依赖的少量上下文
- 删除其余低价值候选项

## 第三阶段：生成完整稿包

在用户确认条目后，不要立刻写入 Purple Sys。先确认 `整理档位` 已锁定，再生成完整稿包，固定包含四类内容：

1. `meeting note draft`
2. `daily 今日沟通与协同` 单条草稿
3. `student log / project log` 追加草稿
4. `拟写入位置`

若 `整理档位` 尚未锁定，先追问，不要直接进入第三阶段。

### Meeting Note Draft

生成 meeting note 时，遵守以下规则：

- 使用当前 Purple Sys meeting template 作为最小脚手架
- 保留 frontmatter 和 H1
- 若已知会议日期，H1 使用真实会议日期
- 只有会议日期明确时，才补 `date:` 字段
- `related_projects` 与 `related_people` 不能稳定判断时，保持空值
- `status` 默认保持 `inbox`

正文结构固定为：

- 开头一段 `总述`
- 主体按模式分组
- 结尾 `后续动作 / 待跟进`

按人模式正文：

- 总述
- 每个关键人一个小节
- `## 后续动作 / 待跟进`

按事模式正文：

- 总述
- 每个关键事项一个小节
- `## 后续动作 / 待跟进`

正文写作要求：

- 使用中文
- 按 `整理档位` 控制压缩强度；在对应档位内，用尽量少的字保留该档位要求的信息
- 不复述对话过程
- 不保留口头语
- 不默认使用表格
- 不默认写“AI 洞察”
- 只在必要时保留少量语境，说明为什么形成某个判断或安排

整理档位对正文的影响：

- `全细节`：尽量保留全部已确认的有效信息点；允许按人 / 按事归并，但不主动拔高成简短结论。
- `详细`：保留绝大多数有效信息；允许合并重复表达，压缩过程性往复，但仍以完整保留为主。
- `标准`：只保留关键判断、决策、动作、节点变化和必要背景；省略多数支撑细节。
- `极简`：只保留主题、结论、下一步和少量必要未决事项；正文应明显短于其他档位。
- 这四档主要作用于 `meeting note draft`；`daily` 与 `student log / project log` 回填草稿仍保持高密度短写，不随档位明显变长。

### Daily 回填草稿

daily 只输出一条高密度 bullet：

- 形式为“与谁沟通了什么，明确了什么”
- 若已有 meeting note 标题，优先链接 meeting note
- 不把完整会议内容塞回 daily

### Student Log / Project Log 回填草稿

只有在存在明确长期对象时才生成。

student log：

- 按日期追加 1-3 条 bullet
- 只保留会改变长期判断、方向或下一步的内容

project log：

- 只保留项目层面的决策、节点变化、依赖或后续动作
- 不复制 meeting note 的叙述

### 拟写入位置

在完整稿包末尾明确说明本轮准备写入哪些位置，例如：

- meeting note：`04 Record/...`
- daily：`02 Periodic/Daily/...`
- student log：`04 Record/log ...`
- project log：`04 Record/log ...`

如果某个相关 log 不存在，不要自动创建；要在这里明确标出“缺失，未写入”。

## 第四阶段：写入 Purple Sys

只有在用户明确给出写入许可后，才进入写入阶段。

可视为写入许可的典型表达包括：

- “可以写入”
- “写进去吧”
- “直接落库”
- “按这个写入 Purple Sys”

进入写入阶段后，默认执行以下动作：

1. 创建或改写本轮 meeting note
2. 把 daily 草稿写入对应日期的 `今日沟通与协同`
3. 把相关 student log / project log 追加到对应日期条目下
4. 返回实际写入位置和未写入位置

除非用户明确要求只写一部分，否则不要只写 meeting note 而漏掉 daily / log。

## Purple Sys 边界

- 这个 skill 服务 `04 Record/` 的 `meeting`，不是 `log` 的替代品。
- 这个 skill 可以给出回填草稿，但不负责决定是否新建长期 log。
- 如果用户明确说可以写入，默认把本轮已经确认的 meeting note、daily 和相关既有 log 一并写回。
- 若相关 student log / project log 不存在，只提示“可挂接对象缺失”，不要自动创建。

## 常见判断

### transcript 很乱时

- 先判断哪些内容是噪音
- 再判断哪些内容值得进入候选清单
- 宁可少收低价值细节，也不要把正式记录写成脏摘要

### speaker 标签明显错误时

- 先利用上下文和已知参会人做初步判断
- 明确区分“高把握判断”和“待用户裁决”
- 关键归属不明时，退回“未确认发言 / 多人讨论”

### 用户只给 transcript 和模式时

- 先自行推断标题、日期、参与人候选
- 若用户已用自然语言表达长短要求，直接映射到对应档位
- 若 `整理档位` 仍不明确，先追问档位
- 仍然先走候选清单阶段，但未锁定档位前不要直接跳正式稿

### 用户给了丰富元信息时

- 不重复追问
- 若仍缺 `整理档位`，补问这一项
- 直接进入候选清单阶段

## 执行方式

在 live vault 中工作时：

- 优先读取 transcript 文件本身，以及已有相关 daily / meeting / log 上下文
- 不为补完所有 metadata 而拖慢整理
- 用户未明确授权写入前，默认只在对话中展示完整稿包
- 用户明确授权后，自动写入 meeting note、daily 和相关既有 log
- 写入后要说明实际落点；若某些 log 缺失，也要说明哪些没有写成

当需要快速查看固定输出格式、字段和示例时，读取 [references/output-contract.md](references/output-contract.md)。
