# 输出契约

## 第一阶段：工作底稿

第一阶段固定输出四块：

1. `元信息草稿`
2. `归属待确认`
3. `候选要点清单`
4. `建议保留项`

建议格式如下。

```md
## 元信息草稿

- 整理档位：待确认
- 模式：按人
- 标题：2026-04-01 EIT 流态化闲聊会
- 日期：2026-04-01
- 时间：待确认
- transcript 质量：中等；噪音较多，speaker 缺失
- 参会人候选：邓正元、吴恺乔、张舒瑞、姚冬旭、谢劲超、刘雨

## 归属待确认

- 00:06:57 后的组织性发言大概率来自老师，但无法仅凭文本稳定区分是邓正元还是吴恺乔。
- 00:07:16-00:07:41 的汇报内容应归到张舒瑞。

## 候选要点清单

### P1 老师

- P1.1 [判断] 当前方向应尽量避免停留在重复性的冷态工作。
- P1.2 [动作] 先做 prototype，再在原型基础上迭代改进。

### P2 张舒瑞

- P2.1 [进展] 博士方向进一步收束到 revolving bed / 转鼓与 AI 结合。
- P2.2 [背景] 最近主要在补 CNN、OpenCV、U-Net 等基础内容。
- P2.3 [动作] 下一步应尽快把已有设备原型、workflow 和 AI 分析结合起来。

## 建议保留项

- 建议保留：P1.1 P1.2 P2.1 P2.3
- 可选保留：P2.2
- 默认删除：与设备、网络、寒暄有关的噪音片段
```

## 候选要点规则

- 每条只表达一个意思。
- 每条尽量压成一句。
- 先做降噪和合并，再写候选项。
- 候选项不是 transcript 摘抄，不要保留明显口头语。

标签固定只有四类：

- `[进展]`
- `[判断]`
- `[动作]`
- `[背景]`

编号规则：

- 按人：`P1.1 / P1.2 / P2.1 ...`
- 按事：`T1.1 / T1.2 / T2.1 ...`

## 第二阶段：用户回复格式

用户不需要写长反馈。可接受的典型回复：

```text
P1 是邓正元，不是吴恺乔。
保留 P1.1 P1.2 P2.1 P2.3，删除其余。
这次按详细写。
张舒瑞这段写短一点，并补一个 student log 草稿。
```

如果用户只给保留编号，则默认：

- 保留这些编号
- 补上少量必要上下文
- 删除其余低价值候选项

## 整理档位契约

`整理档位` 是必选项，没有默认值。

- 若用户直接写 `全细节 / 详细 / 标准 / 极简`，直接采用。
- 若用户用自然语言明确表达长短要求，可直接映射，不必重复追问。
- 若用户只说“这是个很重要的会”，仍然不够，必须继续追问档位。
- 第一阶段可先产出基础候选清单，但进入第三阶段前必须锁定档位。

常见映射：

- “不要漏任何细节”“每一句都重要” -> `全细节`
- “详细一点”“写细一点” -> `详细`
- “正常整理”“常规 meeting note” -> `标准`
- “写短一点”“两三百字”“只要结论和动作” -> `极简`

固定追问模板：

```text
这次你要哪种整理档位：全细节、详细、标准、极简？
全细节=尽量不漏信息；详细=保留大多数有效内容；标准=只留关键判断/动作；极简=只留主题、结论和下一步。
```

## 第三阶段：完整稿包

完整稿包固定四类内容：

1. `meeting note draft`
2. `daily 今日沟通与协同` 单条草稿
3. `student log / project log` 回填草稿（仅在对象明确且 log 已存在时提供）
4. `拟写入位置`

进入第三阶段前，必须锁定 `整理档位`。若档位仍未明确，先追问，不直接生成完整稿包。

type: record
record_type: meeting
status: inbox

related_people: []

### 整理档位对照示例

以下示例都基于同一场讨论。差别只在压缩强度，不在事实本身。

`全细节` 仍然不是 transcript 誊写；它只是尽量完整保留有效信息。

#### 全细节

```md
这次沟通围绕张舒瑞当前方向、前期准备状态和下一步落点展开。讨论重点不在泛泛汇报，而在于把方向进一步收束，并尽快把学习内容和现有原型接起来。

## 张舒瑞

- 当前方向进一步收束到 revolving bed / 转鼓与 AI、图像处理结合，而不是继续停留在泛化的冷态重复工作。
- 最近主要在补 CNN、OpenCV、U-Net 等基础内容，说明前期仍处在补工具和方法栈的阶段。
- 老师强调补基础本身不是目的，下一步应尽快把已有设备原型、workflow 和可得数据结合起来，先做一版 prototype 分析，再决定后续怎么迭代。
- 当前讨论还隐含一个判断：后续工作应尽量朝更有延展性、能形成论文问题的方向推进，而不是只做局部重复验证。
```

#### 标准

```md
这次沟通的重点是进一步收束张舒瑞当前方向，并把讨论拉回到后续可执行动作上。

## 张舒瑞

- 当前方向进一步收束到 revolving bed / 转鼓与 AI、图像处理结合。
- 近期主要在补相关方法基础，下一步应尽快结合已有原型和数据做初步分析。
```

#### 极简

```md
这次主要把张舒瑞的方向收束到转鼓与 AI、图像处理结合，并明确下一步不要只停留在补基础。

## 后续动作 / 待跟进

- 尽快结合现有原型和数据做一版初步分析。
```

## 第四阶段：写入

当用户明确回复“可以写入”“写进去吧”“落库吧”等表达后，默认把第三阶段给出的内容自动写入 Purple Sys。

默认写入顺序：

1. 写 meeting note
2. 写 daily
3. 追加相关 student log / project log
4. 返回写入结果摘要

写入结果摘要建议格式：

```md
已写入：

- `04 Record/20260401 EIT 流态化闲聊会.md`
- `02 Periodic/Daily/2026-04-01.md`
- `04 Record/log 2024 张舒瑞.md`

未写入：

- `04 Record/log 转鼓相关项目.md`：目标 log 不存在，未自动创建
```

## 正式写作要求

- 用中文。
- 长度由 `整理档位` 决定，但都要能独立阅读。
- 保留决定、判断、动作、节点变化和少量必要背景。
- 不写表格，除非用户明确要求。
- 不默认增加“AI 洞察”“总结点评”等额外段落。
- 不把候选清单痕迹带进正式稿。
