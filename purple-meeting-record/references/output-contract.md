# 输出契约

## 第一阶段：工作底稿

第一阶段固定输出四块：

1. `元信息草稿`
2. `归属待确认`
3. `候选要点清单`
4. `建议保留项`

建议格式如下。

```md
## 元信息草稿

- 模式：按人
- 标题：2026-04-01 EIT 流态化闲聊会
- 日期：2026-04-01
- 时间：待确认
- transcript 质量：中等；噪音较多，speaker 缺失
- 参会人候选：邓正元、吴恺乔、张舒瑞、姚冬旭、谢劲超、刘雨

## 归属待确认

- 00:06:57 后的组织性发言大概率来自老师，但无法仅凭文本稳定区分是邓正元还是吴恺乔。
- 00:07:16-00:07:41 的汇报内容应归到张舒瑞。

## 候选要点清单

### P1 老师

- P1.1 [判断] 当前方向应尽量避免停留在重复性的冷态工作。
- P1.2 [动作] 先做 prototype，再在原型基础上迭代改进。

### P2 张舒瑞

- P2.1 [进展] 博士方向进一步收束到 revolving bed / 转鼓与 AI 结合。
- P2.2 [背景] 最近主要在补 CNN、OpenCV、U-Net 等基础内容。
- P2.3 [动作] 下一步应尽快把已有设备原型、workflow 和 AI 分析结合起来。

## 建议保留项

- 建议保留：P1.1 P1.2 P2.1 P2.3
- 可选保留：P2.2
- 默认删除：与设备、网络、寒暄有关的噪音片段
```

## 候选要点规则

- 每条只表达一个意思。
- 每条尽量压成一句。
- 先做降噪和合并，再写候选项。
- 候选项不是 transcript 摘抄，不要保留明显口头语。

标签固定只有四类：

- `[进展]`
- `[判断]`
- `[动作]`
- `[背景]`

编号规则：

- 按人：`P1.1 / P1.2 / P2.1 ...`
- 按事：`T1.1 / T1.2 / T2.1 ...`

## 第二阶段：用户回复格式

用户不需要写长反馈。可接受的典型回复：

```text
P1 是邓正元，不是吴恺乔。
保留 P1.1 P1.2 P2.1 P2.3，删除其余。
张舒瑞这段写短一点，并补一个 student log 草稿。
```

如果用户只给保留编号，则默认：

- 保留这些编号
- 补上少量必要上下文
- 删除其余低价值候选项

## 第三阶段：完整稿包

完整稿包固定四类内容：

1. `meeting note draft`
2. `daily 今日沟通与协同` 单条草稿
3. `student log / project log` 回填草稿（仅在对象明确且 log 已存在时提供）
4. `拟写入位置`

### Meeting Note Draft 形状

```md
---
type: record
record_type: meeting
status: inbox
related_projects: []
related_people: []
created: 2026-04-02
date: 2026-04-01
---

# 2026-04-01 - EIT 流态化闲聊会

这次沟通的重点不是逐条汇报细节，而是进一步收束每位学生当前方向，并把讨论拉回到后续可执行的动作上。

## 老师

- 当前方向应尽量从重复性的冷态工作转向更有延展性的热态、原型和可发表问题。
- 不要一开始就完全另起炉灶，先做 prototype，再在实验和讨论中迭代。

## 张舒瑞

- 当前方向进一步收束到 revolving bed / 转鼓与 AI、图像处理方法结合。
- 最近主要在补机器学习和图像处理基础，后续应尽快结合已有设备原型和数据做初步分析。

## 后续动作 / 待跟进

- 张舒瑞继续补前期基础，并开始准备与转鼓原型和已有 workflow 结合的初步分析。
```

## 第四阶段：写入

当用户明确回复“可以写入”“写进去吧”“落库吧”等表达后，默认把第三阶段给出的内容自动写入 Purple Sys。

默认写入顺序：

1. 写 meeting note
2. 写 daily
3. 追加相关 student log / project log
4. 返回写入结果摘要

写入结果摘要建议格式：

```md
已写入：

- `04 Record/20260401 EIT 流态化闲聊会.md`
- `02 Periodic/Daily/2026-04-01.md`
- `04 Record/log 2024 张舒瑞.md`

未写入：

- `04 Record/log 转鼓相关项目.md`：目标 log 不存在，未自动创建
```

## 正式写作要求

- 用中文。
- 尽量短，但要能独立阅读。
- 保留决定、判断、动作、节点变化和少量必要背景。
- 不写表格，除非用户明确要求。
- 不默认增加“AI 洞察”“总结点评”等额外段落。
- 不把候选清单痕迹带进正式稿。
