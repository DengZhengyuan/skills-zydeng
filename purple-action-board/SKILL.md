# purple-action-board

## 概述

`purple-action-board` 是 Purple Sys 之上的“行动判断层”。

它不创建正式 task 库，不要求用户手工维护待办，也不替代 `purple-periodic`。它的职责是：

- 从 Purple Sys 的事实层提炼行动线程
- 维护不成主线、但仍值得持续看见的小事项
- 回捞曾被记录、后来沉下去、但可能仍值得重新判断的事项
- 在用户需要时判断“现在更该做什么”
- 区分“推进中”“等待条件”“待确认完成”“可忽略”
- 把结果写入固定页面 `00 Dashboard/AI 行动面板.md`

当用户想知道“下一步该做什么”“哪些事情卡住了”“最近哪些重要事项没推进”“更新行动面板”时，应使用本 skill。

## 核心规则

- 这是按需分析系统，不是主动提醒系统。
- 只基于 Purple Sys 自身记录做判断，不创建正式任务库，不要求用户手工维护一套 task。
- 不把 daily 中每个 bullet 直接变成线程；面板以“主线线程 + 小事项”组织，而不是按单条记录组织。
- 默认不扫描整个 vault，必须控制来源范围。
- 重要但暂时推进不了的事项，放入 `等待条件`，不要误判为普通拖延。
- 允许从旧记录中回捞“曾被记录、后来没有继续推进、但可能仍有价值”的事项，但必须先做筛选和归并，不能把零碎记录直接堆成待办。
- Purple Sys 默认全部视为工作相关内容，不再额外区分“工作 / 非工作”。
- 默认启用“候选审阅模式”：先整理候选、再由用户确认、最后才重写面板。
- 默认采用逐条确认、三档推荐、宁多勿漏的取向。
- 对旧条目，`没有新证据` 不能直接等于 `可以放掉`；仍重要的事项应先候选转入 `等待条件`。
- 用户的判断应写回行动面板本身，而不是只停留在对话中。

## 触发范围

以下场景触发：

- “我接下来该做什么”
- “更新行动面板”
- “最近哪些重要事情没推进”
- “帮我判断哪些线程更紧急”
- “基于 Purple Sys 给我下一步建议”
- “有哪些重要但暂停的事别忘了”

以下场景不触发：

- 普通 daily 总结
- weekly / monthly 常规汇总
- 单纯归档 daily 到 log

普通 periodic 工作流仍由 [$purple-periodic](/Users/zydeng/.codex/skills/my-skills/purple-periodic/SKILL.md) 处理。

## 默认输出

- 候选审阅阶段：先在对话中输出候选审阅清单，格式遵循 `references/review-contract.md`
- 固定页面：`00 Dashboard/AI 行动面板.md`
- 默认更新方式：用户确认后整体重算覆盖
- 旧面板只作为参考读取，不做局部拼补

正式行动面板的固定结构与字段契约见 `references/board-contract.md`。候选审阅阶段的输出契约见 `references/review-contract.md`。

## 信息读取规则

默认按四层读取，而不是只看增量：

1. `变化层`
   - 上次行动面板更新之后新增的 `record / material / project log / student log`
   - 用同时间段的 daily / weekly 补上下文
2. `短期背景层`
   - 最近 14 天的 periodic notes
   - 与其明确相关的 project / student / admin logs
3. `中期背景层`
   - 最近 1 个月的 weekly notes
   - 关键 logs
4. `长周期回溯层`
   - 默认不读
   - 用户明确要求时，扩展到最近 3 个月

优先级：

- 旧条目是否被推进：优先看 recent `record / material / project log / student log`
- daily：原始推进与当天事实的补充层
- weekly：阶段性判断、阻塞和重点方向
- material / meeting：只在反复出现或提供关键决策时读取

除“当前线程”外，也允许从较早记录中识别“沉底事项”：

- 事项曾在 daily、weekly、project/log、meeting、material 中被明确记录
- 近期没有新的推进证据
- 但从内容判断，仍可能对后续工作有帮助，或值得用户重新决定去留
- 这类事项默认不直接进入面板，而是先作为“回捞候选”进入判断流程

## 旧条目处置规则

每次更新已有行动面板时，都要先对旧条目做候选分流，而不是直接重写页面。

旧条目候选固定分成 4 类：

- `建议更新保留`
- `建议转等待条件`
- `建议放掉`
- `证据不足，需你判断`

判定原则：

- 只要看到上次面板之后有明确推进证据，优先列入 `建议更新保留`
- 没看到推进证据，但从上下文看仍重要，默认列入 `建议转等待条件`
- 只有在近期证据明显表明 `已完成 / 已失效 / 已被别的线程覆盖` 时，才列入 `建议放掉`
- 如果证据冲突、过旧、或不足以稳定判断，放入 `证据不足，需你判断`
- `没有新证据` 本身不能直接等于 `可以放掉`

## 线程提取规则

面板按“线程”组织，而不是按单条 bullet 组织。

线程示例：

- `Paper 3 收尾`
- `二维床 / CFB 合同与采购`
- `泰格开题准备`
- `刘博异常压降排查`
- `年度预算`
- `某学生当前方向收束`

提取规则：

- 有现成 project/log 时，优先归到对应线程。
- 没有 project/log，但近期反复出现且有明确下一步时，也允许进入候选。
- 只出现一次、没有后续动作的碎片，不进入候选。
- 不要因为一次短期主题，就自动新建 project 或 log。

## 小事项提取规则

除了主线线程外，还允许保留“琐碎但重要的小事项”。

进入小事项区的条件：

- 该事项在 Purple Sys 中被记录过，且当前仍有明确动作意义
- 它值得继续被看见，但暂时不足以形成一条长期主线
- 该事项可能只需要一到两步推进，不一定会反复出现

与线程的分界：

- 主线线程：有稳定主题、会持续跟进、或会牵动多个动作 / 依赖
- 小事项：有明确动作、值得推进，但暂时不构成稳定主线

状态迁移：

- 如果某事项后续反复出现、形成依赖链、或明显牵动主线，应升级为正式线程
- 如果某事项转为外部依赖，应移入 `等待条件`
- 如果某事项已完成或明显失效，应直接移出面板

## 新条目候选提名规则

新条目生成时，不直接给最终面板，而是先给候选池。

候选固定分成两类：

- `新线程候选`
- `新小事项候选`

并按三档推荐等级分层：

- `强推荐`
- `推荐`
- `候补观察`

默认取向是宁多勿漏，但必须靠推荐等级和证据做收束。

推荐等级含义：

- `强推荐`：最近证据强、下一步明确、现在就值得占用注意力
- `推荐`：有动作意义，但优先级略低或仍需用户裁决
- `候补观察`：有价值但证据或时机还不够，优先供用户做“不漏项”检查

提名原则：

- 优先提名 recent 反复出现、带明确下一步、能牵动主线的主题
- 同一主题如果尚不足以成为主线，但确有动作意义，先作为小事项候选而不是直接丢弃
- 沉底事项回捞时，也先进入候选池，不直接落板

## 沉底事项回捞规则

除了当前活跃线程外，还要识别“曾被记录、后来沉下去”的候选事项。

回捞条件：

- 该事项在 Purple Sys 中出现过，且内容可识别，不是完全无法命名的碎片
- 虽然近期没有新证据，但仍可能对后续研究、实验、论文、行政或系统建设有帮助
- AI 无法仅凭现有记录判断它应继续推进、等待条件、已完成还是可忽略

回捞原则：

- 不要求它曾经是高优先级主线；只要“被记录过且可能仍有价值”即可进入候选
- 不把单条 bullet 原样搬上面板；需要先判断它应进入线程候选还是小事项候选
- 明显已经失效、已被后续内容覆盖、或只是随手记录且无后续意义的内容，不必回捞
- 第一次系统梳理时，可以集中回捞一轮；后续更新只挑少量最值得恢复判断的事项

## 候选审阅与确认机制

每次更新已有行动面板前，默认执行两阶段流程：

1. 读取旧面板
2. 读取新增事实和近期背景
3. 先输出旧条目处置候选
4. 再输出新条目候选
5. 等用户逐条确认
6. 根据用户确认结果，整体重算新版面板

首次生成面板时，也使用同样流程，只是不输出“旧条目处置候选”，只输出“新条目候选”。

默认的旧条目可确认动作：

- 更新保留
- 转等待条件
- 放掉
- 暂不动

默认的新条目可确认动作：

- 放入主线程
- 放入小事项
- 放入等待条件
- 本次不放入
- 放弃

如果候选很多，可以按类别展示，但默认仍应支持逐条确认。

## 行动面板状态使用原则

固定状态只有 4 类：

- `推进中`
- `等待条件`
- `待确认完成`
- `可忽略`

判断原则：

- `推进中`：近期仍在推进，或现在值得继续推进
- `等待条件`：重要，但当前受外部依赖、时机、资源或主观判断限制，短期无法推进
- `待确认完成`：从记录看大概率已完成，但信息不足以完全确认
- `可忽略`：不再重要、已失效或暂时不值得占用注意力

额外原则：

- 旧条目近期没有新证据，但仍明显重要时，应优先候选转入 `等待条件`
- `候补观察` 只是候选审阅阶段的推荐等级，不进入正式面板

如果用户明确说“这件事最近做不了，但别忘了”，应放入 `等待条件`，以后每次更新仍继续显示。

## 用户判断的落点

用户对线程的判断默认写回行动面板本身，而不是分散写回原始 logs。

写回规则：

- 最终面板只写入用户确认保留或纳入的内容
- 用户确认 `本次不放入` 的新候选，不进入主板，但要写进 `本次更新时向用户确认的判断`
- 用户确认 `放掉` 的旧条目，不再进入主板，但要写进 `本次更新时向用户确认的判断`
- 用户确认从旧区块迁移到 `等待条件` 的条目，也要写进 `本次更新时向用户确认的判断`
- 默认不新增常驻 `候补观察` 区块，避免面板长期膨胀

## 执行方式

当用户在 live vault 中工作时，优先使用 `obsidian-cli` 读取内容。

典型流程：

1. 先读取现有行动面板；如果不存在，则按首次生成处理。
2. 按四层规则读取所需来源，并优先判断上次面板后的 `record / material / log` 变化。
3. 按 `references/review-contract.md` 产出候选审阅清单，不直接写面板。
4. 等用户逐条确认。
5. 按 `references/board-contract.md` 的固定结构整体重写 `00 Dashboard/AI 行动面板.md`。

## 测试与验收

更新或使用本 skill 时，至少应能稳定识别出以下类型的线程：

- `Paper 3`
- `二维床 / CFB`
- `泰格开题`
- `刘博异常压降`
- `预算`

并将其放入合理区块，而不是简单复述 recent daily。

同时应满足：

- 旧条目在上次面板后有明确 `log / record / material` 推进证据时，应进入 `建议更新保留`
- 旧条目近期无推进，但仍明显重要时，应进入 `建议转等待条件`，而不是直接掉出面板
- 旧条目近期有完成或失效证据时，应进入 `建议放掉`
- 新出现的强信号主题，应进入 `新线程候选 / 强推荐`
- 单步可执行但不构成长期主线的事项，应进入 `新小事项候选`
- 用户确认 `本次不放入` 的新候选，不进主板，但会在摘要中留下本次判断
- 用户确认 `放掉` 的旧条目，不再出现在主板，但会在摘要中留下放掉记录
- 候选列表可以偏长，但最终主板仍应明显比候选清单更收敛，不退化成待办清单
