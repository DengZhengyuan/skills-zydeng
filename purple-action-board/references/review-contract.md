# 行动面板候选审阅契约

该契约用于“候选审阅阶段”的对话输出，而不是最终落入 `00 Dashboard/AI 行动面板.md` 的正式页面。

默认目标：

- 先让 AI 做初步整理和判断
- 再让用户逐条确认
- 最后才按 `board-contract.md` 重写正式面板

## 固定结构

候选审阅阶段默认按以下顺序输出：

1. 标题
2. 一段简短说明
3. `旧条目处置候选`
4. `新线程候选`
5. `新小事项候选`
6. `建议你逐条确认的动作`

首次生成面板时，没有旧面板可参考，可以省略 `旧条目处置候选`。

## 标题与说明

标题建议固定为：

```markdown
# 行动面板候选审阅
```

说明区需要简要写清：

- 本次读取的时间范围
- 这是候选审阅，不是最终面板
- 默认采用逐条确认
- 候选取向是宁多勿漏

## 旧条目处置候选

旧条目处置候选固定分成 4 类：

- `建议更新保留`
- `建议转等待条件`
- `建议放掉`
- `证据不足，需你判断`

每条旧条目候选至少应包含以下字段：

- `条目名`
- `当前所在区块`
- `最近新增证据`
- `AI 建议动作`
- `建议理由`

建议格式：

```markdown
### 条目名

- 当前所在区块：现在值得推进
- 最近新增证据：2026-04-02 组会；log 某学生
- AI 建议动作：建议转等待条件
- 建议理由：最近没有实质推进，但主题仍重要，不能直接放掉
```

## 新条目候选

新条目候选固定分成两类：

- `新线程候选`
- `新小事项候选`

每一类都按三档推荐等级分层：

- `强推荐`
- `推荐`
- `候补观察`

每条新条目候选至少应包含以下字段：

- `候选名`
- `建议放入位置`
- `推荐等级`
- `最近证据`
- `建议下一步`
- `为什么这次值得讨论`

建议格式：

```markdown
### 候选名

- 建议放入位置：现在值得推进
- 推荐等级：强推荐
- 最近证据：2026-04-01 daily；2026-04-02 组会
- 建议下一步：先把问题边界拉清，再安排一轮推进
- 为什么这次值得讨论：最近反复出现，且已经开始牵动主线
```

## 用户可确认的动作

默认的旧条目可确认动作：

- 更新保留
- 转等待条件
- 放掉
- 暂不动

默认的新条目可确认动作：

- 放入主线程
- 放入小事项
- 放入等待条件
- 本次不放入
- 放弃

如果候选较多，可以按类别展示，但必须保留逐条确认的能力。

## 输出边界

- 候选审阅阶段不要直接重写正式面板
- 候选可以偏长，但必须用推荐等级和证据做收束
- `候补观察` 只存在于候选审阅阶段，不进入正式面板
- 对旧条目，`没有新证据` 不能直接等于 `建议放掉`
